Introduction to the iSAQB Certification Program

Introduction to the iSAQB Certification Program

The International Software Architecture Qualification Board (iSAQB) is a globally recognized organization dedicated to standardizing software architecture education and certification. It provides a structured curriculum for professionals to enhance their skills in designing robust, scalable, and maintainable software systems.

The Certified Professional for Software Architecture - Foundation Level (CPSA-F) is the entry-level certification offered by iSAQB. It equips professionals with the foundational knowledge and skills needed to excel in software architecture, covering key areas such as design, communication, and evaluation of architectures.

Basic Terms in Software Architecture

Software architecture involves several fundamental concepts:

  • Architecture: The structure of a system, comprising components, their relationships, and principles guiding design and evolution.
  • Component: A modular, reusable unit with defined interfaces.
  • Stakeholder: Individuals or groups with an interest in the system (e.g., developers, users, management).
  • Quality Attributes: Measurable properties like performance, scalability, and security.

Requirements and Constraints

Requirements define what the system should do, while constraints limit how it can be done.

  • Functional Requirements: Specific features or behaviors (e.g., "The system must process payments").
  • Non-Functional Requirements: Quality attributes (e.g., "The system must handle 1000 users concurrently").
  • Constraints: External limits (e.g., budget, technology stack, regulatory compliance).

Design and Development of Software Architectures

Designing an architecture involves breaking down the system into manageable parts.

  • Top-Down Approach: Start with high-level requirements and refine into components.
  • Patterns: Use established solutions like MVC or Microservices.
  • Iterative Design: Refine the architecture through feedback and prototyping.

0. How Software Architecture Evolves

Software architecture is not a static blueprint—it’s a living, evolving entity shaped by multiple forces and refined over time. Here’s how it unfolds:

Architecture, Organization, and Systems Shape Each Other

The way a company is structured, how teams work, and what systems are in place all influence the software architecture—and vice versa.

Customer and Team Inputs Continuously Evolve

Requirements, goals, and priorities can change as customers and teams learn more or respond to the market.

Architecture Must Adapt Iteratively

Instead of designing everything up front, architecture should evolve step-by-step based on feedback and real-world use.

Spot and Prevent Design Issues Early

Early attention to architecture helps avoid bigger problems later—like performance bottlenecks or inflexible systems.

Start with Customer Needs and Goals

Good design begins by listening to the people using and building the system, ensuring their needs are reflected in the solution.

Rough Design: Overall System Structure

The rough design phase establishes the foundation of the system architecture by defining:

  • System Architecture: Key architectural patterns, styles, and approaches.
  • Subsystems: Major functional areas and their boundaries.
  • Interfaces: Communication points between subsystems and with external systems.
  • Responsibilities: Distribution of system requirements among architectural elements.
  • Technology Selection: Major technology choices and frameworks.

Outcome: A high-level blueprint that addresses major quality attributes and constraints.

Detailed Design: Internal Structure

The detailed design phase elaborates the internals of the architectural elements:

  • Component Design: Internal structure and behavior of individual components.
  • Data Structure Design: Design of domain objects, data models, and persistence schema.
  • Algorithm Design: Specific processing logic and computational approaches.
  • Error Handling: Exception and error management strategies.
  • Resource Management: Memory, storage, and processing resource allocation.

Outcome: Implementation-ready specifications that guide development.

Design principles don’t exist in isolation—they work together to create robust architectures. The following diagram illustrates how these principles interact to solve common architectural challenges:

Single Responsibility Principle (SRP)

A class or component should have only one reason to change.

Open/Closed Principle (OCP)

Entities should be open for extension but closed for modification.

Liskov Substitution Principle (LSP)

Subclasses should be substitutable for their base classes.

Interface Segregation Principle (ISP)

Clients should not depend on interfaces they don’t use.

Dependency Inversion Principle (DIP)

Depend on abstractions, not concretions.

Separation of Concerns (SoC)

Handle different aspects in distinct modules.

Benefits of Applying Design Principles

  • Reduced Complexity: Makes systems easier to understand.
  • Increased Flexibility: Enables adaptation to changes.
  • Improved Changeability: Localizes change impact.
  • Enhanced Maintainability: Simplifies updates.
  • Better Testability: Facilitates testing.
  • Improved Reusability: Enables component reuse.

Description and Communication of Software Architectures

Effective communication ensures all stakeholders understand the architecture.

  • Views: Different perspectives (e.g., logical, physical, deployment).
  • Notations: UML diagrams, box-and-line diagrams.
  • Documentation: Architecture Decision Records (ADRs), technical specs.

Analysis and Evaluation of Software Architectures

Evaluating an architecture ensures it meets requirements and can handle real-world challenges.

  • Architecture Evaluation Methods: Techniques like ATAM (Architecture Tradeoff Analysis Method) to assess quality attributes.
  • Risk Identification: Identify potential risks (e.g., performance bottlenecks) early in the design phase.
  • Scenario-Based Analysis: Test the architecture against specific scenarios (e.g., high user load, system failure).
  • Feedback Loops: Use evaluation results to refine the architecture iteratively.

Tools for Software Architects (Optional)

Tools can enhance architecture design and evaluation.

  • Modeling: Enterprise Architect, ArchiMate
  • Analysis: SonarQube, JMeter
  • Documentation: Confluence, Markdown
